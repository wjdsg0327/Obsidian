## 1.2 Fable 5 / Mythos 5 具体更新了什么

### 1）能力层升级：面向更难的知识工作与代码任务

Anthropic 对 Fable 5 / Mythos 5 的定位不是“小版本优化”，而是新一代 Mythos-class 能力层，重点是：

- 更复杂的长任务；
- 更难的软件工程问题；
- 更强的知识工作能力；
- 视觉、科研、生命科学相关任务；
- 对连续推理、跨步骤执行和复杂任务规划更友好。

这意味着 Anthropic 不是只在“聊天质量”上迭代，而是在往 **agent / 长任务 / 专业工作流** 推。

### 2）访问与商业策略更复杂

官方页面提到 Fable 5 起初对 Pro、Max、Team、seat-based Enterprise 用户有一段“包含使用”的窗口期，但随后会根据容量和 usage credits 调整。

这说明顶级模型的商业模式正在变得更细：

- 普通订阅不一定长期无限包含最强模型；
- 高能力模型可能按 credits 计费；
- 容量不足时，访问策略会动态变化；
- 企业和高价值场景可能优先。

### 3）新增 Mythos-class 数据留存策略

Anthropic 官方提到：对 Fable 5、Mythos 5 以及未来类似或更高能力级别模型，业务客户流量需要 **30 天数据留存**。

官方解释是：

- 不用于训练新 Claude 模型；
- 不用于非安全目的；
- 目的是防御复杂攻击、发现新 jailbreak、降低误报；
- 会记录人类访问日志，并在绝大多数情况下 30 天后删除。

我的理解：这是一个很关键的信号。越强的模型，越需要安全监控；但越多安全监控，就越会和企业隐私、合规、数据主权产生摩擦。

### 4）安全争议：政府要求暂停访问

Anthropic 官方声明称，美国政府要求暂停 Fable 5 / Mythos 5 访问，理由与潜在 jailbreak / 国家安全担忧有关。

Anthropic 的反驳重点大概是：

- 政府提供的是“窄范围、非通用”的潜在 jailbreak 证据；
- Anthropic 认为相关能力并非 Fable 5 独有，其他公开模型也能做类似事情；
- Anthropic 认为如果以这种标准要求召回商业模型，可能会实质性阻碍整个前沿模型行业部署；
- Anthropic 仍主张政府应有能力阻止不安全部署，但流程应透明、公平、技术事实充分。

这个事件比普通产品更新重要，因为它说明：**前沿 AI 模型正在进入类似先进芯片、网络武器、关键基础设施工具的监管逻辑。**

## 1.3 Claude Code `v2.1.181` 更新内容

GitHub Release `v2.1.181` 的主要变化：

### 新增能力

- 新增 `/config key=value` 语法：
  - 可以直接在 prompt 里设置任意 setting；
  - 示例：`/config thinking=false`；
  - 支持 interactive、`-p` 和 Remote Control。

- 新增 `sandbox.allowAppleEvents`：
  - macOS 上允许沙箱命令发送 Apple Events；
  - 是 opt-in 设置，默认不随便放开。

- 新增 `CLAUDE_CLIENT_PRESENCE_FILE` 环境变量：
  - 指向一个 marker file；
  - 当用户在电脑前时，可以抑制移动端 push 通知。

### 体验优化

- 捆绑 Bun runtime 升级到 **1.4**。
- 长段落 streaming 改进：现在文本会逐行出现，不再等到第一个换行才显示。
- API 连接在 thinking 中途断开时自动重试，不再直接显示 “Connection closed while thinking”。
- subagent 面板改进：
  - idle subagents 30 秒后自动隐藏；
  - 列表最多显示 5 行；
  - footer 里有键盘提示。
- MCP OAuth 浏览器页面视觉风格更像 Claude Code，并且授权成功后自动关闭。

### 行为调整

- 全屏模式下打开 URL 改为需要 Cmd+click（macOS）或 Ctrl+click，更接近原生终端行为。
- `Improved N memories` 默认不再列出具体文件，除非 verbose 模式。

### 为什么重要

这一版不是“模型能力”更新，而是把 Claude Code 做得更像成熟开发工具：配置可脚本化、远程控制更顺、断线恢复更稳、subagent 面板更可用。

## 2.2 Codex 产品层更新：从“写代码”变成“每个角色的工作工具”

OpenAI 6 月 2 日发布 **Codex for every role, tool, and workflow**。

官方标题强调：

- New role-specific plugins；
- Sites；
- annotations；
- 帮助团队用 Codex 做更多事。

### 具体变化

#### 1）Role-specific plugins

Codex 不再只是通用 coding agent，而是可以针对不同角色和工作流配置插件。

这意味着：

- 开发者可以有开发插件；
- 产品、运营、知识工作者可以有不同插件；
- 企业内部可以围绕自己的工具链定制 Codex。

#### 2）Sites

Sites 的方向是把 Codex 产出的东西变成可分享、可访问的页面或工作成果。

价值在于：

- 不只是生成代码；
- 可以把结果交付成一个团队可看的站点；
- 更适合设计评审、文档、demo、内部工具。

#### 3）Annotations

Annotations 用于对 Codex 工作结果进行更细粒度反馈。

这对 agent 很关键，因为 agent 长期工作的核心问题不是“能不能生成”，而是“怎么被人类审阅、纠偏、持续改进”。

## 2.3 OpenAI 收购 Ona：给 Codex 一个“长期工作的地方”

OpenAI 6 月 11 日宣布收购 **Ona**。

官方标题：**Expands Codex with secure, customer-controlled cloud infrastructure for long-running agents across software and knowledge work.**

重点解读：

- Codex 需要一个安全、客户可控的云环境；
- 这个环境要支持长期运行 agent；
- 不只面向软件开发，也面向知识工作；
- 典型任务包括运行测试、解决 issue、现代化应用、处理漏洞、支持复杂流程。

我的判断：这一步非常重要。OpenAI 正在补 Codex 的“执行环境”。没有长期、隔离、可审计、客户可控的执行环境，agent 只能停留在聊天框和一次性脚本；有了执行环境，才可能变成企业里的持续 worker。

# 7. 给老王的实用建议

## 如果只是日常用 AI

优先关注：

- ChatGPT / Codex 是否更适合写代码；
- Gemini 3.5 Flash 在搜索和移动端是否更快；
- Claude Fable 5 是否恢复稳定访问；
- 订阅计划是否开始把最强模型改成 credit 计费。

## 如果要搭本地 AI / Agent 环境

优先关注：

- CLIProxyAPI 是否升级到 `v7.2.16` 或更高；
- OpenClaw 是否保持在 `v2026.6.8` 或更新；
- Hermes Agent 的桌面端和中文界面是否适合试用；
- Claude Code / Codex CLI 是否与 WSL2、Windows Terminal、远程 session 配合稳定。

## 如果要选编程 Agent

我的判断：

- **Claude Code**：更适合长上下文、复杂项目、交互式工程协作；最近修复对 WSL2 和远程 session 友好。
- **OpenAI Codex**：更像企业级长期 agent 平台，特别是配合 AWS/Oracle/Ona 后，适合有云治理需求的团队。
- **Gemini / Antigravity**：更适合低成本、高速度、Google 生态和多模态入口。
- **OpenClaw / Hermes**：更适合自己掌控工作流、消息渠道、工具、skills、长期任务。
