Rust Axum 资料和教程

这是一份面向 Rust 已入门用户的 Axum 后端开发笔记：目标不是背 API，而是理解 Axum 的工作方式，并能写出一个结构清楚、能接数据库、能处理错误、能部署的 Web 服务。

适合：Rust 基础之后稳定线：Axum 0.8.x运行时：Tokio生态：Tower / Hyper

01. Axum 是什么

Axum 是 Tokio 团队维护的 Rust HTTP 路由与请求处理库。它的定位很清楚：提供舒服的路由、请求提取、响应转换和错误处理，但不重新发明一套中间件系统，而是直接接入 tower 生态。

Axum 在 Rust Web 栈里的位置

Axum 的核心特点

  • 无宏路由：不像某些框架那样大量用属性宏，Axum 通过 Router::new().route(...) 组合路由。
  • Extractor：用类型来声明你要从请求里取什么，比如 Path、Query、Json、State。
  • IntoResponse：handler 返回的东西只要能转成 HTTP 响应即可。
  • Tower 生态：CORS、Trace、Timeout、Compression、Limit 等都可以通过 tower-http 组合。
常见误区
Axum 不是“全家桶框架”。它不替你规定 ORM、配置系统、目录结构、认证方案。好处是灵活，坏处是新手需要自己组合生态。

一句话记住：Axum = Router + Handler + Extractor + IntoResponse + Tower。

02. 环境与最小服务

建议使用稳定 Rust、Tokio 和 Axum 0.8.x。Axum 主分支可能已经朝 0.9 发展，但学习和项目落地优先看 crates.io/docs.rs 的稳定版。

cargo new axum-demo
cd axum-demo
cargo add axum
cargo add tokio --features full
cargo add serde --features derive
cargo add serde_json
cargo add tracing tracing-subscriber

最小服务：

use axum::{routing::get, Router};

#[tokio::main]
async fn main() {
    tracing_subscriber::fmt::init();

    let app = Router::new()
        .route("/", get(root));

    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000")
        .await
        .unwrap();

    axum::serve(listener, app).await.unwrap();
}

async fn root() -> &'static str {
    "Hello, Axum!"
}
最小服务启动过程

验证

cargo run
curl http://127.0.0.1:3000/
常见错误
如果 #[tokio::main] 报错，通常是 Tokio 没开 macros 和运行时特性。新手直接用 tokio --features full 最省心。

03. 路由与 Handler

Axum 的路由由 Router 组合，handler 是普通异步函数。你可以按 HTTP 方法绑定不同处理函数。

use axum::{routing::{get, post}, Router};

let app = Router::new()
    .route("/", get(index))
    .route("/users", get(list_users).post(create_user))
    .route("/users/{id}", get(get_user));

async fn index() -> &'static str { "ok" }
async fn list_users() {}
async fn create_user() {}
async fn get_user() {}
Router 把路径 + 方法分发给 handler

路由组织建议

  • 小项目：直接在 main.rs 里写路由。
  • 中型项目：拆成 routes/user.rs、routes/auth.rs。
  • 大型项目：按领域拆分，比如 modules/users、modules/orders。
pub fn routes() -> Router<AppState> {
    Router::new()
        .route("/users", get(list_users).post(create_user))
        .route("/users/{id}", get(get_user))
}
常见误区
不要把业务逻辑全塞进 handler。handler 最好只负责“取请求、调用 service、转响应”。真正逻辑放到 service 层更容易测试。

04. Extractor 请求提取器

Extractor 是 Axum 最重要的概念之一。handler 参数不是随便写的，而是告诉 Axum：“我要从请求里提取这些东西”。

Extractor用途例子
Path<T>路径参数/users/{id}
Query<T>查询参数?page=1
Json<T>JSON 请求体POST body
State<T>应用共享状态数据库连接池
HeaderMap请求头Authorization
use axum::{extract::{Path, Query}, Json};
use serde::{Deserialize, Serialize};

#[derive(Deserialize)]
struct Page {
    page: Option<u32>,
}

#[derive(Deserialize)]
struct CreateUser {
    username: String,
}

#[derive(Serialize)]
struct User {
    id: u64,
    username: String,
}

async fn get_user(Path(id): Path<u64>) -> String {
    format!("user id = {id}")
}

async fn list_users(Query(page): Query<Page>) -> String {
    format!("page = {:?}", page.page)
}

async fn create_user(Json(input): Json<CreateUser>) -> Json<User> {
    Json(User { id: 1, username: input.username })
}
请求被 Extractor 拆成 handler 参数
参数顺序提醒
会消费 body 的 extractor，比如 Json，通常放在最后。因为请求体只能被读取一次。

05. Response 响应设计

Axum 的 handler 返回值只要实现 IntoResponse，就能变成 HTTP 响应。常见返回方式：

use axum::{http::StatusCode, response::IntoResponse, Json};
use serde::Serialize;

#[derive(Serialize)]
struct ApiResp<T> {
    code: i32,
    message: String,
    data: T,
}

async fn plain() -> &'static str {
    "ok"
}

async fn created() -> impl IntoResponse {
    (StatusCode::CREATED, "created")
}

async fn json_data() -> impl IntoResponse {
    Json(ApiResp {
        code: 0,
        message: "success".into(),
        data: vec!["axum", "tokio"],
    })
}
IntoResponse 把不同返回值统一成 HTTP 响应

建议统一 API 响应格式

业务项目中建议统一 JSON 结构，比如：

{
  "code": 0,
  "message": "success",
  "data": {}
}
常见误区
不要每个接口随手返回不同 JSON 结构。前端、调用方和日志排查都会很痛苦。

06. 共享状态与项目结构

实际项目通常需要共享数据库连接池、配置、缓存客户端等。Axum 推荐用 State。

use axum::{extract::State, routing::get, Router};
use std::sync::Arc;

#[derive(Clone)]
struct AppState {
    app_name: String,
}

async fn health(State(state): State<Arc<AppState>>) -> String {
    format!("{} is ok", state.app_name)
}

let state = Arc::new(AppState { app_name: "demo".into() });
let app = Router::new()
    .route("/health", get(health))
    .with_state(state);
共享状态在每个请求中被廉价 clone

推荐目录结构

src/
  main.rs              # 启动入口
  app.rs               # Router 组装
  state.rs             # AppState
  config.rs            # 配置
  error.rs             # AppError
  modules/
    users/
      mod.rs
      routes.rs
      handler.rs
      service.rs
      model.rs
实用原则
Router 层只负责把 URL 接到 handler；handler 负责 HTTP 适配；service 负责业务逻辑；repository/dao 负责数据库。

07. 错误处理

Axum 的错误处理推荐做法：定义自己的 AppError，实现 IntoResponse，handler 返回 Result<T, AppError>。

use axum::{http::StatusCode, response::{IntoResponse, Response}, Json};
use serde_json::json;

#[derive(Debug)]
enum AppError {
    NotFound,
    BadRequest(String),
    Internal(String),
}

impl IntoResponse for AppError {
    fn into_response(self) -> Response {
        let (status, message) = match self {
            AppError::NotFound => (StatusCode::NOT_FOUND, "not found".to_string()),
            AppError::BadRequest(msg) => (StatusCode::BAD_REQUEST, msg),
            AppError::Internal(msg) => (StatusCode::INTERNAL_SERVER_ERROR, msg),
        };

        (status, Json(json!({
            "code": status.as_u16(),
            "message": message,
            "data": null
        }))).into_response()
    }
}

async fn get_user() -> Result<Json<serde_json::Value>, AppError> {
    Err(AppError::NotFound)
}
业务错误转换成 HTTP 响应
常见误区
不要在 handler 里到处 unwrap()。Web 服务里 panic 会让请求失败，严重时影响服务稳定性。

08. 中间件与 Tower

Axum 自己不搞一套独立中间件体系，而是使用 Tower。常用中间件通常来自 tower-http。

cargo add tower-http --features trace,cors,compression-full,timeout
use axum::Router;
use tower_http::{
    trace::TraceLayer,
    cors::CorsLayer,
    compression::CompressionLayer,
};
use std::time::Duration;
use tower::ServiceBuilder;

let middleware = ServiceBuilder::new()
    .layer(TraceLayer::new_for_http())
    .layer(CorsLayer::permissive())
    .layer(CompressionLayer::new());

let app = Router::new()
    .route("/", axum::routing::get(|| async { "ok" }))
    .layer(middleware);
请求穿过中间件链再进入路由

什么时候用中间件

  • 统一日志：TraceLayer。
  • 跨域：CorsLayer。
  • 压缩：CompressionLayer。
  • 认证：可以自定义 middleware 或 extractor。
  • 请求 ID、超时、限流：Tower/Tower HTTP 组合。
安全提醒
生产环境不要无脑 CorsLayer::permissive()，应该限制允许的域名、方法和请求头。

09. 数据库、认证与文件

Axum 不绑定数据库。常见组合是 sqlx + PostgreSQL/MySQL/SQLite。连接池放进 AppState，handler 通过 State 拿。

cargo add sqlx --features runtime-tokio-rustls,postgres,macros
cargo add uuid chrono
use axum::{extract::State, Json};
use sqlx::PgPool;
use std::sync::Arc;

#[derive(Clone)]
struct AppState {
    db: PgPool,
}

async fn list_users(State(state): State<Arc<AppState>>) -> Result<Json<Vec<User>>, AppError> {
    let users = sqlx::query_as!(User, "select id, username from users")
        .fetch_all(&state.db)
        .await
        .map_err(|e| AppError::Internal(e.to_string()))?;

    Ok(Json(users))
}

认证思路

认证可以分为两层：

  • 解析身份：从 Header/Cookie 取 token，校验 JWT 或 session。
  • 注入身份：把当前用户放进 request extensions，handler 通过自定义 extractor 拿。
典型后端请求链路

文件上传

Axum 支持 multipart，需要启用 feature：

cargo add axum --features multipart
use axum::extract::Multipart;

async fn upload(mut multipart: Multipart) {
    while let Some(field) = multipart.next_field().await.unwrap() {
        let name = field.name().unwrap_or("").to_string();
        let bytes = field.bytes().await.unwrap();
        println!("field={name}, size={}", bytes.len());
    }
}
常见误区
上传文件不要直接无限读入内存。生产项目要限制大小，必要时流式写入磁盘或对象存储。

10. 部署、测试与练习项目

优雅关闭

服务部署时要处理 Ctrl+C 或容器停止信号，让请求有机会正常结束。

let listener = tokio::net::TcpListener::bind("0.0.0.0:3000").await.unwrap();
axum::serve(listener, app)
    .with_graceful_shutdown(shutdown_signal())
    .await
    .unwrap();

async fn shutdown_signal() {
    tokio::signal::ctrl_c().await.expect("failed to install Ctrl+C handler");
}

测试 handler

Axum 应用本质上也是 Tower Service，可以用 tower::ServiceExt 发送测试请求。

cargo add tower --features util --dev
cargo add http-body-util --dev
#[tokio::test]
async fn health_works() {
    use axum::{body::Body, http::{Request, StatusCode}};
    use tower::ServiceExt;

    let app = Router::new().route("/health", get(|| async { "ok" }));

    let response = app
        .oneshot(Request::builder().uri("/health").body(Body::empty()).unwrap())
        .await
        .unwrap();

    assert_eq!(response.status(), StatusCode::OK);
}
学习路线：从 Demo 到可部署服务

练习项目顺序

  1. 健康检查服务：GET /health 返回版本、时间、状态。
  2. Todo API：实现 CRUD，先内存 HashMap，再换 SQLite/PostgreSQL。
  3. 用户系统：注册、登录、JWT、当前用户接口。
  4. 文件上传：上传图片，限制大小，返回访问 URL。
  5. 管理后台 API：分页、筛选、统一错误、日志、权限。

生产部署清单

  • 监听 0.0.0.0:端口，前面用 Nginx/Caddy 反代。
  • 配置从环境变量读取，不要写死密钥。
  • 打开结构化日志和请求 ID。
  • 数据库连接池设置合理上限。
  • 开启 graceful shutdown。
  • Docker 镜像用多阶段构建，最终镜像尽量小。
学习建议
先不要急着追求复杂架构。Axum 最值得掌握的是类型化请求处理：Extractor 负责输入，IntoResponse 负责输出，AppError 负责错误，State 负责共享资源。把这四个练熟，项目就稳了。
