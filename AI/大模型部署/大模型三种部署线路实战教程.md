# 大模型三种部署线路实战教程

整理时间：2026-06-03

## 总览

大模型部署通常可以分成三条主线：

1. **本地直跑**：模型直接在本机或服务器进程里加载运行。
2. **服务化部署**：把模型部署成 HTTP / OpenAI-compatible API 服务，对外提供推理接口。
3. **云端/API 托管**：直接调用第三方厂商或云平台的大模型 API。

这三条线路不是互斥的。实际项目里常见路径是：

```text
本地直跑验证效果 -> 服务化部署做应用集成 -> 云/API 或自建集群做生产弹性
```

---

## 1. 线路一：本地直跑

### 1.1 适合场景

本地直跑适合：

- 学习和实验。
- 单机测试模型效果。
- 做 LoRA / 微调后的快速验证。
- 离线环境，不希望数据出机器。
- 只需要脚本调用，不需要多人并发访问。

### 1.2 常见工具

| 工具 | 说明 | 适合模型 |
|---|---|---|
| Transformers | Hugging Face 官方生态，通用性强 | 大多数 HF 模型 |
| llama.cpp | CPU / 低显存 / GGUF 量化模型 | Llama、Qwen、Mistral 等 GGUF |
| Ollama | 封装友好，适合桌面和快速体验 | 常见开源模型 |
| LM Studio | 图形化，本地聊天体验好 | GGUF / 本地模型 |

### 1.3 Transformers 本地直跑示例

安装：

```bash
pip install torch transformers accelerate
```

Python 示例：

```python
from transformers import AutoTokenizer, AutoModelForCausalLM
import torch

model_name = "Qwen/Qwen2.5-7B-Instruct"

tokenizer = AutoTokenizer.from_pretrained(model_name, trust_remote_code=True)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.bfloat16,
    device_map="auto",
    trust_remote_code=True,
)

messages = [
    {"role": "system", "content": "你是一个有帮助的助手。"},
    {"role": "user", "content": "解释一下什么是 RAG。"},
]

prompt = tokenizer.apply_chat_template(
    messages,
    tokenize=False,
    add_generation_prompt=True,
)

inputs = tokenizer(prompt, return_tensors="pt").to(model.device)

outputs = model.generate(
    **inputs,
    max_new_tokens=512,
    temperature=0.7,
    top_p=0.9,
)

print(tokenizer.decode(outputs[0][inputs.input_ids.shape[-1]:], skip_special_tokens=True))
```

### 1.4 llama.cpp 本地直跑示例

适合 GGUF 量化模型。

安装或编译 llama.cpp 后：

```bash
./llama-cli \
  -m ./models/qwen2.5-7b-instruct-q4_k_m.gguf \
  -p "请解释一下什么是向量数据库" \
  -n 512
```

### 1.5 Ollama 本地直跑示例

安装 Ollama 后：

```bash
ollama pull qwen2.5:7b
ollama run qwen2.5:7b
```

命令行调用：

```bash
ollama run qwen2.5:7b "写一个 Python 读取 CSV 的例子"
```

### 1.6 优点

- 上手快。
- 不需要搭 API 服务。
- 数据不出本机。
- 方便调试模型、prompt、量化版本。

### 1.7 缺点

- 并发能力弱。
- 应用集成不方便。
- 进程管理、监控、限流都要自己做。
- 多人使用或生产环境不适合直接脚本跑。

### 1.8 推荐用法

本地直跑主要用于：

```text
验证模型能不能用、效果好不好、显存够不够、prompt 怎么写
```

不要一开始就把它当生产方案。

## 2. 线路二：服务化部署

### 2.1 适合场景

服务化部署适合：

- 应用需要通过 HTTP 调用模型。
- 多个客户端共用一个模型服务。
- 需要并发、队列、吞吐优化。
- 要接入 LangChain、Dify、OpenWebUI、RAG 系统、评测系统。
- 希望接口兼容 OpenAI API。

### 2.2 常见工具

| 工具 | 特点 | 适合场景 |
|---|---|---|
| vLLM | 高吞吐，PagedAttention，OpenAI-compatible API | 生产推理、并发服务 |
| SGLang | 推理框架，适合复杂 agent / structured generation | 高性能推理、复杂流程 |
| TGI | Hugging Face Text Generation Inference | HF 生态、企业部署 |
| Ollama API | 简单好用，本地服务 | 个人/小团队快速部署 |
| llama.cpp server | 轻量，适合 GGUF | CPU/量化模型服务 |
| FastAPI + Transformers | 自定义程度高 | 特殊逻辑、实验服务 |

### 2.3 vLLM OpenAI-compatible API 部署

安装：

```bash
pip install vllm
```

启动服务：

```bash
vllm serve Qwen/Qwen2.5-7B-Instruct \
  --host 0.0.0.0 \
  --port 8000 \
  --dtype bfloat16 \
  --trust-remote-code
```

测试：

```bash
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen/Qwen2.5-7B-Instruct",
    "messages": [
      {"role": "user", "content": "介绍一下 Milvus 是什么"}
    ],
    "temperature": 0.7,
    "max_tokens": 512
  }'
```

Python 调用：

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="EMPTY",
)

resp = client.chat.completions.create(
    model="Qwen/Qwen2.5-7B-Instruct",
    messages=[{"role": "user", "content": "写一个 Docker Compose 示例"}],
    temperature=0.7,
)

print(resp.choices[0].message.content)
```

### 2.4 Docker 部署 vLLM 示例

```bash
docker run --gpus all \
  -p 8000:8000 \
  --ipc=host \
  vllm/vllm-openai:latest \
  --model Qwen/Qwen2.5-7B-Instruct \
  --dtype bfloat16 \
  --trust-remote-code
```

如果要挂载模型缓存，建议按老王约定放到 Docker 数据目录：

```bash
docker run --gpus all \
  -p 8000:8000 \
  --ipc=host \
  -v /mnt/d/work/docker/vllm/huggingface:/root/.cache/huggingface \
  vllm/vllm-openai:latest \
  --model Qwen/Qwen2.5-7B-Instruct \
  --dtype bfloat16 \
  --trust-remote-code
```

### 2.5 llama.cpp server 部署

### 2.6 Ollama API 服务

Ollama 默认会提供本地 API。

拉模型：

```bash
ollama pull qwen2.5:7b
```

调用：

```bash
curl http://localhost:11434/api/chat \
  -d '{
    "model": "qwen2.5:7b",
    "messages": [
      {"role":"user", "content":"解释一下 RAG 的流程"}
    ],
    "stream": false
  }'
```

OpenAI 兼容接口通常是：

```text
http://localhost:11434/v1
```

### 2.7 服务化部署关键参数

| 参数 | 作用 |
|---|---|
| `max_model_len` / context length | 上下文长度 |
| `gpu_memory_utilization` | GPU 显存使用比例 |
| `tensor_parallel_size` | 多卡张量并行 |
| `max_num_seqs` | 最大并发序列数 |
| `dtype` | float16 / bfloat16 / float32 |
| `quantization` | 量化方式 |
| `trust_remote_code` | 允许加载模型自定义代码 |

vLLM 示例：

```bash
vllm serve Qwen/Qwen2.5-14B-Instruct \
  --host 0.0.0.0 \
  --port 8000 \
  --dtype bfloat16 \
  --tensor-parallel-size 2 \
  --gpu-memory-utilization 0.9 \
  --max-model-len 32768 \
  --trust-remote-code
```

### 2.8 优点

- 接口标准，方便应用集成。
- 可以做并发、限流、监控。
- 多个应用可以共用同一个模型服务。
- vLLM / TGI / SGLang 吞吐高，适合生产。

### 2.9 缺点

- 部署复杂度比本地直跑高。
- 需要处理服务稳定性、GPU 资源、日志、鉴权。
- 模型服务暴露到网络时有安全风险。

### 2.10 推荐用法

服务化部署适合真正做应用：

```text
RAG 系统、智能客服、代码助手、企业内网知识库、Agent 服务
```

如果是生产环境，建议至少加：

- 反向代理：Nginx / Caddy。
- API Key 鉴权。
- 日志与监控。
- 请求超时。
- 并发限制。
- Docker Compose 或 systemd 管理进程。

## 3. 线路三：云端/API 托管

### 3.1 适合场景

云端/API 托管适合：

- 不想自己买 GPU / 管服务器。
- 快速接入强模型。
- 模型能力优先，部署成本次要。
- 需要稳定 SLA、弹性扩容。
- 项目早期快速验证产品。

### 3.2 常见提供方

| 类型 | 示例 |
|---|---|
| 国际 API | OpenAI、Anthropic、Google Gemini、Mistral |
| 国内 API | 通义千问、DeepSeek、智谱、豆包、Moonshot、百度千帆 |
| 云平台托管 | 阿里云 PAI、腾讯云 TI、AWS Bedrock、Azure OpenAI、Google Vertex AI |
| Serverless GPU | Modal、RunPod、Replicate、Together AI 等 |

### 3.3 OpenAI SDK 调用示例

```bash
pip install openai
```

```python
from openai import OpenAI

client = OpenAI(api_key="你的 API Key")

resp = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "user", "content": "总结一下大模型部署方式"}
    ],
)

print(resp.choices[0].message.content)
```

### 3.4 OpenAI-compatible 第三方 API 调用

很多厂商支持 OpenAI SDK，只需要换 `base_url`。

```python
from openai import OpenAI

client = OpenAI(
    api_key="你的 API Key",
    base_url="https://api.example.com/v1",
)

resp = client.chat.completions.create(
    model="provider-model-name",
    messages=[{"role": "user", "content": "写一个 Milvus 入门说明"}],
)

print(resp.choices[0].message.content)
```

### 3.5 优点

- 上手最快。
- 不需要本地 GPU。
- 通常稳定性和模型能力更强。
- 适合产品快速验证。
- 可直接用强模型、多模态模型、长上下文模型。

### 3.6 缺点

- 数据会发到第三方，需要考虑隐私和合规。
- 成本按 token 或请求计费，量大后可能贵。
- 受限于厂商限流、接口变更、服务可用性。
- 私有化和深度定制能力有限。

### 3.7 推荐用法

云/API 适合：

```text
先把应用做出来、验证业务价值、需要强模型能力但暂时不想管基础设施
```

如果涉及隐私数据，需要提前确认：

- 数据是否允许出网。
- API 厂商是否会训练使用请求数据。
- 是否需要私有化部署或专有云。
- 是否需要日志脱敏。

## 4. 三条线路对比

| 维度 | 本地直跑 | 服务化部署 | 云端/API 托管 |
|---|---|---|---|
| 上手速度 | 快 | 中等 | 最快 |
| 生产可用性 | 低 | 高 | 高 |
| 并发能力 | 弱 | 强 | 强 |
| 成本 | 需要本地硬件 | 需要 GPU 服务器 | 按量付费 |
| 数据隐私 | 最好 | 好，取决于部署位置 | 取决于供应商 |
| 可控性 | 高 | 很高 | 中等或低 |
| 运维复杂度 | 低 | 中高 | 低 |
| 适合阶段 | 学习/实验 | 应用/生产 | 快速产品化 |

## 5. 怎么选

### 5.1 个人学习

推荐：

```text
Ollama / LM Studio / llama.cpp
```

理由：简单，不需要复杂部署。

### 5.2 Python 实验和模型评测

推荐：

```text
Transformers + lm-eval
```

理由：生态完整，方便控制模型和数据。

### 5.3 RAG 知识库 / 企业内网助手

推荐：

```text
vLLM / SGLang / TGI + OpenAI-compatible API
```

理由：应用接入方便，吞吐高，可私有化。

### 5.4 CPU 或低显存机器

推荐：

```text
llama.cpp + GGUF 量化模型
Ollama
```

理由：量化模型对硬件要求低。

### 5.5 快速做产品 Demo

推荐：

```text
OpenAI / DeepSeek / 通义千问 / 智谱 / Moonshot API
```

理由：不用管部署，直接做业务。

### 5.6 正式生产

推荐路线：

```text
先 API 验证业务 -> 再评估成本和隐私 -> 必要时迁移到 vLLM 私有化部署
```

## 6. 实战推荐组合

### 6.1 本地学习组合

```text
Ollama + OpenWebUI
```

特点：像本地 ChatGPT，适合日常使用和学习。

### 6.2 私有知识库组合

```text
vLLM + Qwen/DeepSeek 开源模型 + FastAPI/LangChain/LlamaIndex + Milvus
```

特点：适合企业内网 RAG。

### 6.3 轻量服务器组合

```text
llama.cpp server + GGUF 量化模型 + Nginx
```

特点：成本低，性能有限，但够做简单问答。

### 6.4 生产高吞吐组合

```text
vLLM + 多 GPU + OpenAI-compatible API + 反向代理 + 监控
```

特点：适合多用户并发。

### 6.5 快速 SaaS 组合

```text
OpenAI-compatible API + 业务后端 + 用量统计 + 缓存
```

特点：最快上线，后续再考虑私有化。

## 7. Docker Compose 示例：vLLM 服务

按老王 Docker 数据目录约定，持久化缓存放到：

```text
D:\work\docker\vllm
WSL 路径：/mnt/d/work/docker/vllm
```

`docker-compose.yml`：

```yaml
services:
  vllm:
    image: vllm/vllm-openai:latest
    container_name: vllm-qwen
    restart: unless-stopped
    ports:
      - "8000:8000"
    ipc: host
    volumes:
      - /mnt/d/work/docker/vllm/huggingface:/root/.cache/huggingface
    environment:
      - HF_TOKEN=${HF_TOKEN:-}
    command:
      - --model
      - Qwen/Qwen2.5-7B-Instruct
      - --dtype
      - bfloat16
      - --trust-remote-code
      - --host
      - 0.0.0.0
      - --port
      - "8000"
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]
```

启动：

```bash
docker compose up -d
```

查看日志：

```bash
docker logs -f vllm-qwen
```

测试：

```bash
curl http://localhost:8000/v1/models
```

## 8. 部署前检查清单

### 8.1 模型选择

- 模型大小：7B、14B、32B、70B。
- 上下文长度：8K、32K、128K。
- 是否 instruct/chat 模型。
- 是否支持工具调用、JSON 输出、多模态。
- 是否允许商业使用。

### 8.2 硬件资源

- GPU 型号。
- 显存大小。
- CPU / 内存。
- 磁盘空间。
- 网络带宽。

粗略显存参考：

| 模型 | FP16/BF16 粗略显存 | 4bit 量化粗略显存 |
|---|---:|---:|
| 7B | 14GB+ | 5GB+ |
| 14B | 28GB+ | 9GB+ |
| 32B | 64GB+ | 20GB+ |
| 70B | 140GB+ | 40GB+ |

实际还要考虑 KV cache、batch size、上下文长度。

### 8.3 服务安全

- 不要裸露到公网。
- 加 API Key。
- 加访问日志。
- 加请求大小限制。
- 加超时。
- 加并发限制。
- 敏感数据脱敏。

### 8.4 成本

- GPU 机器月租。
- API token 费用。
- 模型下载和存储。
- 运维时间。
- 高峰并发冗余。

---

## 9. 一句话总结

- **本地直跑**：适合学习、测试、验证模型。
- **服务化部署**：适合应用集成和生产私有化。
- **云端/API 托管**：适合快速上线和使用强模型。

最实用的路线：

```text
先用 API 或本地直跑验证需求，再用 vLLM / SGLang 做服务化私有部署。
```
