# Go：Windows 内存管理专题

Go 自己有 GC 和运行时内存管理；只有在需要调用 Windows 底层能力时，才直接碰 Windows 内存 API。常见场景包括：共享内存、文件映射、调用 DLL、与 C 结构体交互、查询系统内存、调试/读取其他进程内存。

> 官方入口：
> - Go x/sys/windows：https://pkg.go.dev/golang.org/x/sys/windows
> - Windows Memory Management Functions：https://learn.microsoft.com/windows/win32/memory/memory-management-functions
> - Virtual Memory Functions：https://learn.microsoft.com/windows/win32/memory/virtual-memory-functions
> - Heap Functions：https://learn.microsoft.com/windows/win32/memory/heap-functions

## 1. Go 与 Windows 内存 API 的关系

| 需求 | Go 推荐 | Windows API |
|---|---|---|
| 普通内存 | slice、map、struct、`make/new` | 不需要 |
| C/Windows 结构体 | Go struct + `unsafe` | 按 Windows SDK 布局传入 |
| UTF-16 字符串 | `windows.UTF16PtrFromString` | `LPCWSTR` / `LPWSTR` |
| 系统内存状态 | `x/sys/windows` 或 LazyDLL | `GlobalMemoryStatusEx` |
| 虚拟内存 | `windows.VirtualAlloc` / LazyDLL | `VirtualAlloc` / `VirtualFree` |
| 页面权限 | `windows.VirtualProtect` / LazyDLL | `VirtualProtect` |
| 共享内存 | `CreateFileMapping` + `MapViewOfFile` | 文件映射 API |
| 读写其他进程 | `ReadProcessMemory` / `WriteProcessMemory` | 调试/自动化场景 |
| 安全清零 | 手写循环 + `runtime.KeepAlive` 或 DLL | `SecureZeroMemory` |

## 2. Go 调 Windows API 的注意点

### GC 与指针生命周期

Go 的 GC 会管理 Go 对象。把 Go 指针传给 Windows API 时，要保证调用期间对象仍然存活：

```go
buf := make([]byte, 4096)
// 把 &buf[0] 传给 Windows API
// 调用结束后：
runtime.KeepAlive(buf)
```

不要把 Go 指针长期交给 Windows 保存，除非你非常清楚生命周期和 cgo/unsafe 规则。

### `uintptr` 不是安全指针

`uintptr` 只是整数，不会让 GC 保持对象存活。只在 syscall 调用边界临时转换。

### UTF-16 字符串

```go
p, err := windows.UTF16PtrFromString("Local\\MySharedMemory")
if err != nil { panic(err) }
```

### 结构体大小

很多 Windows API 要求结构体字段 `cbSize` / `dwLength`：

```go
s.Length = uint32(unsafe.Sizeof(s))
```

## 3. 常用 API 速查

| 分类 | API | Go 方式 |
|---|---|---|
| 系统内存状态 | `GlobalMemoryStatusEx` | `windows.GlobalMemoryStatusEx` 若版本支持；否则 LazyDLL |
| 虚拟内存 | `VirtualAlloc` / `VirtualFree` | `windows.VirtualAlloc` / `windows.VirtualFree` 或 LazyDLL |
| 页面保护 | `VirtualProtect` / `VirtualQuery` | `windows.VirtualProtect` / LazyDLL |
| 文件映射 | `CreateFileMapping` / `MapViewOfFile` | `windows.CreateFileMapping` / `windows.MapViewOfFile` 或 LazyDLL |
| 共享内存 | `OpenFileMapping` | `windows.OpenFileMapping` 或 LazyDLL |
| 进程内存 | `ReadProcessMemory` / `WriteProcessMemory` | `windows.ReadProcessMemory` / LazyDLL |
| 堆 | `HeapAlloc` / `HeapFree` | 通常不建议，除非与 C ABI 配套 |
| 安全清零 | `SecureZeroMemory` | LazyDLL 或谨慎手写 |

## 4. 案例：查询系统内存状态

如果你当前 `x/sys/windows` 版本有 `GlobalMemoryStatusEx` 和 `MemoryStatusEx`，可直接用。否则用 LazyDLL。

```go
//go:build windows
package main

import (
    "fmt"
    "syscall"
    "unsafe"
)

type memoryStatusEx struct {
    Length               uint32
    MemoryLoad           uint32
    TotalPhys            uint64
    AvailPhys            uint64
    TotalPageFile        uint64
    AvailPageFile        uint64
    TotalVirtual         uint64
    AvailVirtual         uint64
    AvailExtendedVirtual uint64
}

var (
    kernel32 = syscall.NewLazyDLL("kernel32.dll")
    globalMemoryStatusEx = kernel32.NewProc("GlobalMemoryStatusEx")
)

func main() {
    var m memoryStatusEx
    m.Length = uint32(unsafe.Sizeof(m))

    r1, _, err := globalMemoryStatusEx.Call(uintptr(unsafe.Pointer(&m)))
    if r1 == 0 {
        panic(err)
    }

    fmt.Printf("memory load: %d%%\n", m.MemoryLoad)
    fmt.Printf("total phys: %d MB\n", m.TotalPhys/1024/1024)
    fmt.Printf("avail phys: %d MB\n", m.AvailPhys/1024/1024)
}
```

## 5. 案例：VirtualAlloc / VirtualFree

```go
//go:build windows
package main

import (
    "fmt"
    "unsafe"

    "golang.org/x/sys/windows"
)

func main() {
    const size = 4096
    addr, err := windows.VirtualAlloc(0, size, windows.MEM_RESERVE|windows.MEM_COMMIT, windows.PAGE_READWRITE)
    if err != nil {
        panic(err)
    }

    buf := unsafe.Slice((*byte)(unsafe.Pointer(addr)), size)
    buf[0] = 0x41
    buf[1] = 0x42
    fmt.Println(buf[:2])

    err = windows.VirtualFree(addr, 0, windows.MEM_RELEASE)
    if err != nil {
        panic(err)
    }
}
```

> 注意：不同版本的 `x/sys/windows` 函数签名可能略有差别；如果编译不通过，查 `pkg.go.dev/golang.org/x/sys/windows` 当前版本签名。

## 6. 案例：DLL 动态调用 VirtualAlloc

当 `x/sys/windows` 未封装某 API，可用 `NewLazySystemDLL`。

```go
//go:build windows
package main

import (
    "fmt"
    "golang.org/x/sys/windows"
)

var (
    kernel32 = windows.NewLazySystemDLL("kernel32.dll")
    procGetLargePageMinimum = kernel32.NewProc("GetLargePageMinimum")
)

func main() {
    r1, _, _ := procGetLargePageMinimum.Call()
    fmt.Println("large page minimum:", r1)
}
```

## 7. 案例：共享内存思路

Go 里共享内存仍然是 Windows 文件映射：

1. `CreateFileMapping(INVALID_HANDLE_VALUE, ...)` 创建基于 page file 的命名映射。
2. `MapViewOfFile` 得到地址。
3. 用 `unsafe.Slice` 包成 Go slice。
4. 读写内存。
5. `UnmapViewOfFile`，然后 `CloseHandle`。

伪代码：

```go
// name := windows.UTF16PtrFromString("Local\\MySharedMemory")
// h, err := windows.CreateFileMapping(windows.InvalidHandle, nil, windows.PAGE_READWRITE, 0, 4096, name)
// addr, err := windows.MapViewOfFile(h, windows.FILE_MAP_ALL_ACCESS, 0, 0, 4096)
// buf := unsafe.Slice((*byte)(unsafe.Pointer(addr)), 4096)
// buf[0] = 1
// windows.UnmapViewOfFile(addr)
// windows.CloseHandle(h)
```

实际代码要按你当前 `x/sys/windows` 的函数签名微调。

## 8. 读写其他进程内存

典型流程：

1. `OpenProcess(PROCESS_VM_READ | PROCESS_QUERY_INFORMATION, false, pid)`。
2. `ReadProcessMemory` 读取目标地址。
3. `CloseHandle`。

写内存还需要：

- `PROCESS_VM_OPERATION`
- `PROCESS_VM_WRITE`
- 必要时 `VirtualProtectEx`

这类操作常被安全软件拦截，且涉及权限边界。调试器、内存分析器可以用；不要拿来乱注入。

## 9. 什么时候不要直接用 Windows 内存 API

| 需求 | 更好的 Go 方式 |
|---|---|
| 普通缓存 | `make([]byte, n)` |
| 动态对象 | struct/slice/map |
| 文件读写 | `os.ReadFile`、`os.File`、`bufio` |
| 大文件顺序处理 | 流式读取，不一定需要 mmap |
| 加密密钥 | 专门安全库 + 尽量缩短生命周期 |
| HTTP/网络 | 标准库 `net/http` |

## 10. 常见坑

- 不要把 Go 指针长期存进 Windows 侧回调或异步结构里。
- `unsafe.Slice` 的长度必须严格等于映射/分配大小，越界就是崩。
- `VirtualFree(..., MEM_RELEASE)` 时 size 必须为 0。
- `MapViewOfFile` 得到的地址要 `UnmapViewOfFile`，映射句柄要 `CloseHandle`。
- Windows API 返回值不能统一按 Go `err` 理解，有些 API 返回 0 是成功，有些返回非 0 是成功。
- `SecureZeroMemory` 这种防优化语义，Go 普通清零不一定等价；敏感数据要特别处理。
