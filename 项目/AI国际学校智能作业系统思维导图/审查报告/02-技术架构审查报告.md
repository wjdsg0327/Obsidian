# 02-技术架构审查报告

## 1. 总体评分

**评分：78 / 100**

技术文档已经有数据库设计、建表 SQL、接口草案、后端模块、AI 服务、部署运维方案。但当前还存在“关系图有表、SQL 没表”“接口字段粒度不一致”“文件/通知/发布范围缺少正式表”的问题。整体可作为草案，但进入开发前需要统一数据模型。

## 2. 主要技术问题

### 2.1 SQL 草案缺少部分关系图中出现的表

数据库关系图和设计说明中提到：

- `teacher_quality_stats`
- `class_compare_stats`

但 SQL 草案中缺失这两张表。

### 2.2 作业发布范围设计不足

需求支持：

- 全班学生。
- 指定学生分组。
- 个别学生。

但 SQL 只有 `assignments.publish_scope`，缺少明细表。建议新增：

- `assignment_targets`

### 2.3 多教师任课关系缺失

班级只有 `head_teacher_id`，不能表达多个任课教师。建议新增：

- `class_teachers`

### 2.4 文件上传缺少统一文件表

学生提交、PDF、课件、录音、题库导入都涉及文件。当前只用 URL 字段，不利于权限、生命周期、归档和审计。

建议新增：

- `files`

### 2.5 通知缺少数据模型

文档中多次提到作业提醒、批改完成、限额提醒，但没有通知表和接口。

建议新增：

- `notifications`

### 2.6 密码安全表述需要升级

原需求写 SHA-256 + 盐。建议正式文档改为：

- Argon2id 优先。
- bcrypt 可接受。
- 不建议手写 SHA-256 + salt。

## 3. 表结构 / 接口冲突

| 问题 | 表现 | 建议 |
|---|---|---|
| 学生只属于一个 class_id | 国际学校可能按学科分班 | MVP 可保留，后续用 class_students |
| assignments.publish_scope 粒度不足 | 无法指定学生/分组 | 增加 assignment_targets |
| file_urls JSONB 粗糙 | 文件权限和审计难 | 增加 files 表 |
| AI 调用日志有设计但接口未体现 | 成本统计无法查询 | 增加 AI 成本查询接口 |
| 报告导出缺少 report_jobs | PDF 异步生成不好追踪 | MVP 可用 files + task status 简化 |

## 4. 性能与安全风险

1. 学情统计如果实时聚合，数据量大时会慢，需要定时汇总表。
2. 批量 AI 批改必须异步队列化，不能同步阻塞接口。
3. 文件上传应走对象存储或分片上传，不能全部经应用内存中转。
4. 数据权限必须后端强制，不能只靠前端菜单。
5. AI 请求需尽量匿名化学生姓名。
6. 数据导出必须审计。
